## Why this matters

Every discovered `SKILL.md`'s YAML frontmatter is parsed with the deprecated
`serde_yaml` crate. `Cargo.lock` resolves it to
`version = "0.9.34+deprecated"` â€” the upstream maintainer tags the final
release explicitly as deprecated and archived in favor of the maintained
fork `serde_yml`. A deprecated, unmaintained parser owns the input
boundary for workspace and global skill content (user-influenced YAML
under `<workspace>/.agents/skills/` and the user's global data directory).
Any future YAML CVE or parsing bug will not be patched upstream.

Migration is small (single call site, thick test coverage), and the
successor crate has a near-identical API.

## Steps

### Step 1: Swap the dependency

In `src-tauri/Cargo.toml`, replace `serde_yaml = "0.9"` with
`serde_yml = "<recent stable major.minor, currently 0.0.x>"` (the
maintained crate's API is near-identical to `serde_yaml` 0.9; the version
pin should follow the repo's `"MAJOR.MINOR"` style â€” pin to the current
minor at execution time).

Remove the `serde_yaml` line and let cargo resolve `serde_yml` (you will
re-add the line in the lockfile in the next step). Do not leave both
listed â€” `serde_yaml` must be removed entirely.

### Step 2: Update the import + call site

In `src-tauri/src/skills.rs:340` (and any other `serde_yaml::` reference
in the file â€” verify with `rg "serde_yaml" src-tauri/src`):

- Replace `serde_yaml::from_str(frontmatter_str)` with
  `serde_yml::from_str(frontmatter_str)`.
- Replace any `use serde_yaml;` / `use serde_yaml::...` import with the
  equivalent `serde_yml` import (the function name `from_str` is identical).
- Map the new crate's error type to the existing `SkillParseError`
  (or whatever the file already maps `serde_yaml::Error` into â€” match the
  existing `From` impl shape). `serde_yml`'s error type has a near-

If the error type does NOT map cleanly (different `Display` or expectant
fields), add a small `From<serde_yml::Error> for SkillParseError` impl

## STOP conditions

  migration is larger than this plan assumed; report the list before
  completing the swap).

  string AND the new wording diverges AND the test's intent is to assert
  the literal wording (vs. the canonicalized `SkillParseError` message) â€”
  STOP and ask whether to update the assertion to the canonicalized

  report the divergence; this plan assumed a near drop-in swap which is
